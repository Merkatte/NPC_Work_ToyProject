# Commit To NPC Work ToyProject

Use this skill only when one of these is true:

- The user explicitly invokes `commit-to-npc-work-toyproject`.
- Another explicitly invoked skill says to use `commit-to-npc-work-toyproject`.

Do not use this skill automatically. Do not commit or push merely because files changed, a task finished, or the user asks for unrelated development work.

## Target Repository

Push commits to:

```text
https://github.com/Merkatte/NPC_Work_ToyProject
```

Use this remote URL:

```text
https://github.com/Merkatte/NPC_Work_ToyProject.git
```

## Safety Rules

- Do not run `git add`, `git commit`, or `git push` unless the user explicitly requested committing/pushing in the current turn or this skill was explicitly invoked for that purpose.
- Commit only first-party source code directly authored or modified by the user or Codex for this project, plus the Unity `.meta` files required for those source files and source folders.
- Never commit installed packages, imported libraries, SDKs, plugins, vendor code, or any files supplied or generated by them. This prohibition applies even when those files use source-code extensions such as `.cs`.
- Never commit package binaries, debug symbols, documentation, settings, generated modules, or dependency metadata. Examples include `.dll`, `.pdb`, `.mdb`, vendor `.xml`, package readmes, `Assets/Plugins`, `Assets/ThirdParty`, `Assets/Vendor`, `Packages`, and package-created settings such as `Assets/Resources/DOTweenSettings.asset`.
- Leave excluded dependency files unstaged and report that collaborators must install the dependency separately.
- Do not commit prefabs, scenes, images, audio, generated art, documentation, or other non-code project content unless the user explicitly requests those files in the current turn. This exception never permits package or library files.
- Never include Unity generated folders such as `Library`, `Temp`, `Logs`, `obj`, or IDE files in commits.
- Respect existing `.gitignore`.
- Check `git status --short` before staging.
- Review staged files before committing.
- If the working tree contains unrelated user changes, do not revert them. Ask before including them if the requested commit scope is unclear.
- If file authorship or package provenance is uncertain, do not stage the file; ask the user first.

## Workflow

1. Check repository state:

```powershell
git status --short
git remote -v
git branch --show-current
```

2. Ensure the remote URL points to the target repository. If no remote exists, add:

```powershell
git remote add origin https://github.com/Merkatte/NPC_Work_ToyProject.git
```

If `origin` exists but points elsewhere, ask before changing it.

3. Build an explicit allowlist of directly authored first-party source files. Stage exact paths from that allowlist. Do not use broad staging commands such as `git add -A`, `git add .`, or `git add Assets` because they can capture installed dependencies.

4. Review the staged paths before committing:

```powershell
git diff --cached --name-only
git diff --cached --stat
```

Unstage any package, library, vendor, generated, binary, or otherwise disallowed file. Confirm that no installed dependency files remain staged.

5. Commit with a concise message describing the actual source-code change.

6. Push to `origin main` unless the user explicitly asks for another branch.

7. Report the commit hash, push result, and any dependency files intentionally left local.
