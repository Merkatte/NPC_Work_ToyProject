## 1. Planning Summary

현재 프로젝트는 Worker 행동 실행 구조, Farmer의 기본 생존 행동, 밀 생산·운반 판단, 이동·작업 애니메이션과 Cook selector 골격까지 구현된 기술 기반 단계다. 그러나 운반한 밀이 실제 창고 재고에 저장되지 않아 첫 생산 루프가 아직 완결되지 않았다. 모집, 성장, 경제, 전투, 치료와 운영 UI도 구현 전이다.

따라서 Cook 행동이나 신규 직업을 바로 확장하지 않는다. 먼저 첫 수직 슬라이스에 필요한 미결정 기획을 확정하고 `ARCHITECTURE.md`를 작성한 뒤, 기존 Farmer 생산 루프를 실제 재고와 화면에서 검증 가능한 상태로 완성한다. 이후 음식 순환, 성장, 경제·모집, 방어·치료 순으로 확장한다.

### Blocker

`ARCHITECTURE.md`가 존재하지 않는다. 다음 항목의 소유권과 데이터 흐름을 아키텍처에서 확정하기 전에는 관련 기능을 구현하지 않는다.

- 창고·식당·골드·아이템 재고의 소유자
- 일반 주민과 직업별 selector/action/context 경계
- 직업·스탯·경험치·모집 후보 데이터의 저장 방식
- 전투 상태, 부상, 사망과 치료 상태의 전환 책임
- UI가 런타임 상태를 읽는 인터페이스
- 날짜, 상인 방문과 웨이브 진행의 시간 기준

## 3. Dependency Map

```text
첫 수직 슬라이스 기획 결정
  → ARCHITECTURE.md
    → 창고 재고 영속화
      → Farmer 생산 루프 Play Mode 검증
        → 공용 물품·재고 구조
          → 음식 생산·식당 재고
            → Cook 행동
              → 운영 UI와 디버그

스탯 데이터
  → 업무 성과
  → 직업 경험치·레벨
  → 모집 비용
  → 네임드 주민
  → 전직·특화 업무

창고·물품·골드
  → 상인 거래
  → 지속 가능한 모집

전투와 웨이브
  → 부상·전투 불능·사망
  → 병원·치료소

전체 욕구 해결 수단
  → 불쾌 수치
  → 폭동
```

Cook 구현은 창고·물품·음식 재고의 소유권이 확정되고 저장이 가능한 뒤에 시작한다. 병원은 전투 피해 상태가 구현된 뒤에 시작한다. 폭동은 모든 기본 욕구의 증가·해결 흐름이 완성된 뒤에 시작한다.

## 6. Definition of Done

### General

- 관련 SPEC 요구사항과 Acceptance Criteria를 만족한다.
- 책임과 의존 방향이 `ARCHITECTURE.md`, `ProjectStructure.md`, `CodeConvention.md`를 위반하지 않는다.
- 조정 수치를 action 또는 selector에 하드코딩하지 않는다.
- 필수 참조 누락 시 처리되지 않은 예외나 무한 반복 없이 식별 가능한 오류를 남긴다.
- 명령행 C# 빌드 또는 Unity 컴파일이 오류 없이 완료된다.
- 런타임 행동은 Unity Play Mode에서 검증한다.
- 완료 내용, 검증 결과와 남은 문제를 `PROGRESS.md`에 기록한다.

### Feature-Specific

- 재고 기능은 생산·운반·보관·소비·판매 전후 총량을 검증한다.
- AI 행동은 시작, 성공, 실패와 취소 시 상태와 애니메이션을 정리한다.
- 경제 기능은 골드와 재고를 하나의 거래 단위로 원자적으로 갱신한다.
- 모집 기능은 후보 중복 합류와 골드 중복 차감을 방지한다.
- 전투 상태는 부상·전투 불능·사망을 모순되게 동시에 적용하지 않는다.
- 치료 기능은 시설 점유와 주민의 업무 가능 상태를 함께 검증한다.

## 7. QA Checkpoints

| Checkpoint | 검증 범위 | 통과 조건 |
|---|---|---|
| QA-M0 | Farmer 생산·이동·입고, 목적지 누락, 애니메이션 취소 | 정상 루프 반복, 재고 보존, 안전 실패, 콘솔 오류 없음 |
| QA-M1 | 밀·음식 재고, Cook 생산, 식당 소비, 상태 표시 | 생산부터 소비까지 수량 정합성과 화면 상태 일치 |
| QA-M2 | 스탯, 경험치, 상인, 모집, 네임드 | 고정 시드 재현, 거래 정합성, 모집 비용과 합류 규칙 통과 |
| QA-M3 | 웨이브, 표적, 성곽·시청, 피해 상태, 치료 | 승리·패배·피해·치료의 모든 주요 분기 통과 |
| QA-M4 | 전직, 욕구, 폭동, 성능과 회귀 | 전체 Acceptance Criteria와 목표 성능 통과 |

## 8. Risks and Mitigations

| Risk | Trigger | Mitigation | Owner |
|---|---|---|---|
| 기능별 재고가 따로 생겨 정합성이 깨짐 | 창고, 식당, 상인이 각자 독립 수량을 임의 관리 | 아키텍처에서 재고 소유권과 거래 경계를 먼저 확정 | Architecture owner |
| Farmer 구조를 Cook·병사에 복사 | 새 역할이 Farmer의 ActionSet/context를 직접 재사용 | 공용 capability와 역할별 행동 소유권을 분리 | Implementation owner |
| UI가 런타임 구현에 강결합 | UI가 action/selector의 구체 타입을 직접 조회 | 상태 조회 인터페이스 또는 읽기 모델을 아키텍처에 정의 | Architecture owner |
| 미정 기획을 코드가 사실상 확정 | `TBD` 수치를 구현자가 임의 지정 | 관련 Q-ID를 먼저 결정하거나 임시값임을 데이터와 문서에 명시 | Product owner |
| 수직 슬라이스 없이 기능만 증가 | Cook, 전투, 모집 골격이 동시에 늘어남 | 각 Slice의 QA 통과 전 다음 Slice 착수 금지 | Project owner |
| Play Mode 검증 없이 빌드 성공만 확인 | Unity 직렬화·Tween·씬 연결 결함 누락 | 모든 런타임 Task에 Play Mode 체크포인트 유지 | QA owner |

## 9. Runner Instructions

1. `In Progress` 작업이 있으면 새 작업보다 먼저 완료하거나 명시적으로 중단 사유를 기록한다.
2. 의존성이 완료된 가장 낮은 번호의 P0 작업을 선택한다. P0가 없으면 P1, 그다음 P2를 선택한다.
3. `ARCHITECTURE.md` 승인 전에는 `TASK-003` 이후 구현 작업을 시작하지 않는다.
4. 작업 시작 시 `PROGRESS.md`의 `In Progress`에 Task ID, 시작일과 현재 단계를 기록한다.
5. 작업 중 SPEC의 `TBD` 또는 Open Question에 의존하게 되면 임의로 결정하지 말고 작업을 중단하고 질문 ID를 Blocker에 기록한다.
6. 완료 시 Deliverable과 Verification을 모두 충족한 뒤 `Completed Tasks`와 `Verification Performed`를 갱신한다.
7. Slice의 마지막 작업이 끝나면 대응 QA Checkpoint를 통과하기 전 다음 Slice로 진행하지 않는다.
8. 범위 밖 리팩터링과 낮은 우선순위 정리는 현재 Task의 완료 조건에 직접 필요하지 않으면 별도 작업으로 분리한다.

## 10. Change Control

- 게임 규칙, 승패 조건, 상태 전환 또는 플레이어 권한이 바뀌면 먼저 `PublicMD/SPEC.md`를 갱신한다.
- 시스템 책임, 데이터 소유권, 의존 방향 또는 런타임 흐름이 바뀌면 먼저 `ARCHITECTURE.md`를 갱신한다.
- 요구사항이나 아키텍처 변경으로 작업 의존성이 달라지면 기존 Task ID를 유지한 채 이 문서의 Dependencies와 Verification을 갱신한다.
- 완료된 요구사항을 삭제하지 않는다. 폐기 시 SPEC에서 deprecated 사유를 남기고 PLAN에서 대체 또는 제외 근거를 기록한다.
- 새 기능은 기존 Slice의 QA를 우회하여 삽입하지 않는다. 긴급 결함은 P0로 추가하고 영향받는 QA Checkpoint를 다시 실행한다.

### Deferred Coverage

- 구체적인 농사·훈련 구역, 전체 제작 레시피, 보스·적 종류, 건설·철거·수리, 저장/불러오기, 튜토리얼, 난이도 선택과 멀티플레이는 현재 SPEC의 Out of Scope이므로 본 계획에서 제외한다.
- REQ-F-025~026의 전직 전용 제작과 성장 시설은 `TASK-015`까지 명시적으로 연기한다.
- REQ-F-016~019의 전체 욕구·폭동은 기본 음식 순환이 검증된 뒤 `TASK-019`에서 처리한다.
- REQ-NF-001의 목표 개체 수와 프레임률은 SPEC 수치가 `TBD`이므로 `TASK-020` 전까지 측정 기준 확정이 필요하다.
